What’s the evolution of Hikma’s sales between 2025 and 2026?
user query → answer in 5 steps, ≈ 2.7 seconds
1
Query Enhancerquery_enhancer.py
detected_kpis: ["evolution"] · year_p0: 2025 year_p1: 2026
A lightweight LLM reads the raw question and tags it: which business metric is being asked for, which periods to compare. It doesn't generate any SQL yet it lays the groundwork so the next step knows what to inject into the prompt.
2
Prompt Builderprompt_builder.py
schema + catalog + KPI-1 “Evolution” addon + MR filter
Based on the tags from step 1, the builder assembles a tailored prompt: only the schema of the relevant tables, the column catalog, and the SQL “instructions” specific to the “Evolution” KPI. Less noise in the context means fewer errors at the next step.
3
Agentic Loopllm_service.py
Fixed budget of 3 attempts. On each turn, the model writes SQL, runs it, and reads the result (or the error) itself before deciding what to do next. This is what lets it self-correct without human intervention.
1 · Faulty SQL
2 · Corrected SQL
3 · Final answer
attempt 1 → sqlite3.OperationalError: no such column: vente_ht
agent → re-reads the column catalog injected at step 2, spots that the correct column is called montant_ht
attempt 2 → rewrites the query with the correct column → executes successfully
The agent doesn't stop at the first execution error: it treats the SQLite error message as an observation, adjusts its query, and tries again exactly like an analyst fixing a typo in a column name. Without this safety net, the smallest naming mistake would make the whole answer fail.
4
SQL Executiondatabase.py · SQLite
hikma_2025 = 4,812,340 TND → hikma_2026 = 5,436,900 TND
The query corrected in step 3 is first sanitized (sanitize_sqlite_sql, against injections and destructive commands), then run against the real data this result feeds the final answer.
